# Editorial Example

一个展示 Lynx 双线程架构的文本排版演示，对比 Hybrid 和 Pure MTS 两种模式的性能差异。

## 文件结构

| 文件名 | Entry (Bundle) | 架构模式 | 说明 |
|--------|----------------|----------|------|
| `main.tsx` | `main` | Hybrid (MTS + BTS) | MTS 处理动画/触摸，BTS 处理文本布局和渲染 |
| `editorial-engine-mts.tsx` | `mts-only` | Pure MTS | 全部逻辑在主线程，零跨线程通信 |

## 架构对比

### Hybrid (`main.tsx`)
```
┌─────────────────────────────────────────────────────┐
│                    Main Thread                       │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐  │
│  │ 物理模拟    │  │ 触摸处理    │  │ FPS 追踪    │  │
│  └─────────────┘  └─────────────┘  └─────────────┘  │
│                          │                          │
│              runOnBackground(setOrbs)               │
│                          ▼                          │
└─────────────────────────────────────────────────────┘
                           │
┌─────────────────────────────────────────────────────┐
│                  Background Thread                   │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐  │
│  │ React 渲染  │  │ 文本布局    │  │ JSX 构建    │  │
│  └─────────────┘  └─────────────┘  └─────────────┘  │
└─────────────────────────────────────────────────────┘
```

**特点：**
- MTS 专注高优先级任务（物理、触摸）
- BTS 利用 React 的批量更新和虚拟 DOM diff
- 每帧一次 `runOnBackground` 同步数据
- 1-2 帧延迟，但整体吞吐量更高

### Pure MTS (`editorial-engine-mts.tsx`)
```
┌─────────────────────────────────────────────────────┐
│                    Main Thread                       │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐  │
│  │ 物理模拟    │  │ 触摸处理    │  │ FPS 追踪    │  │
│  └─────────────┘  └─────────────┘  └─────────────┘  │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐  │
│  │ 文本布局    │  │ DOM 操作    │  │ 元素池管理  │  │
│  └─────────────┘  └─────────────┘  └─────────────┘  │
│                                                      │
│  ← 共享模块 (prepareWithSegments) →                  │
└─────────────────────────────────────────────────────┘
```

**特点：**
- 零跨线程通信延迟
- 使用共享模块 (`with { runtime: 'shared' }`) 在 MTS 调用 `prepareWithSegments`
- 元素池模式：预创建固定数量的元素，通过 `setStyleProperties` 更新
- 单线程瓶颈：所有计算必须在 16ms 内完成

## 性能对比

| 指标 | Hybrid | Pure MTS |
|------|--------|----------|
| 触摸响应延迟 | ~0ms (MTS 直接处理) | ~0ms (MTS 直接处理) |
| 文本更新延迟 | 1-2 帧 | 0 帧 |
| 复杂布局吞吐量 | 更高 (React 批量更新) | 较低 (逐个 DOM 操作) |
| 内存占用 | 较高 (虚拟 DOM) | 较低 (元素池) |
| 适用场景 | 复杂 UI、大量元素 | 简单 UI、低延迟要求 |

## 如何选择

**选择 Hybrid 当：**
- UI 复杂度高（多列、Drop Cap、Pullquote 等）
- 元素数量动态变化
- 需要 React 生态支持

**选择 Pure MTS 当：**
- 需要最低延迟
- UI 结构简单、元素数量固定
- 愿意手动管理 DOM 更新

## 运行

```bash
# 开发模式
pnpm dev

# 构建
pnpm build
```

访问 `/main.lynx.bundle` 查看 Hybrid 模式，访问 `/mts-only.lynx.bundle` 查看 Pure MTS 模式。
